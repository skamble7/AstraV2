About these diagrams

Skills (sk.*) and capabilities (cap.*) coexist in ASTRA. Neither replaces the other. A new Electron desktop app serves as the frontend for the skill path, powered by the Astra Agent (TypeScript, in-process). The VSCode extension continues unchanged, handling capabilities only. Artifact kinds (cam.*), workspace management, and artifact storage are shared between both paths.

1. Coexistence Model

Skills and capabilities are parallel execution primitives sharing the same backend infrastructure for artifacts, workspaces, and runs.

3. Astra Agent — Unified Model

The Astra Agent runs inside the Electron app. Two plan acquisition strategies feed into a single shared execution core. All sk.* identifiers — never cap.*.

The Astra Agent only ever resolves sk.* identifiers. cap.* identifiers are invisible to it. Both run modes stream conversational tokens via the Anthropic SDK SSE.

4. Intent-Driven Run

User expresses intent. Agent loads sk.* skill manifest and uses Claude to select and sequence skills as tools.

intent-run-sequence.mermaid
sequenceDiagram participant UI as Electron UI participant AG as Astra Agent participant SR as skill-registry-service participant AN as Anthropic API participant MCP as MCP Server participant WM as workspace-manager-service participant SS as session-svc UI->>AG: user message (intent) AG->>SR: GET /manifest (sk.* skills as tool defs) SR-->>AG: skill manifest AG->>SS: GET /sessions/{id} SS-->>AG: conversation history AG->>AN: messages.stream({ tools: sk.* skills, messages }) AN-->>AG: SSE token stream AG-->>UI: stream tokens to UI AN-->>AG: tool_use — sk.x.y.z selected AG->>SR: GET /skills/sk.x.y.z (load SKILL.md) SR-->>AG: skill + execution block Note over AG,UI: Approve gate — user confirms inputs AG-->>UI: show approve form (prefilled_inputs) UI->>AG: confirmed inputs AG->>MCP: invoke tool_name (from execution block) MCP-->>AG: artifact result AG-->>UI: stream continues AG->>WM: upsert artifact (cam.* kind) AG->>SS: persist session history AG-->>UI: run complete
Skills are presented as Anthropic tool definitions. The agent selects sk.* skills from the manifest — cap.* identifiers never appear. Artifacts are stored in workspace-manager-service using the shared cam.* artifact kind registry.

5. Pack-Driven Run

User selects a Skill Pack. The plan is read directly from the sk.* playbook — deterministic, no LLM reasoning for plan acquisition.

pack-run-sequence.mermaid
sequenceDiagram participant UI as Electron UI participant AG as Astra Agent participant SR as skill-registry-service participant AN as Anthropic API participant MCP as MCP Server participant WM as workspace-manager-service UI->>AG: user selects Skill Pack AG->>SR: GET /skill-packs/{id}/resolved SR-->>AG: pack + sk.* playbook steps AG->>MCP: tools/list (resolve input schema) MCP-->>AG: input schema + prefilled_inputs AG-->>UI: show approve form UI->>AG: confirmed inputs Note over AG: Plan = playbook steps (no LLM reasoning) loop For each sk.* step in playbook AG->>SR: GET /skills/{sk.id} SR-->>AG: SKILL.md + execution block AG->>AN: stream (input resolution + narration) AN-->>AG: tokens AG-->>UI: stream tokens AG->>MCP: invoke tool_name MCP-->>AG: artifact result AG->>AN: enrichment (diagram + narrative) AN-->>AG: tokens AG-->>UI: stream continues end AG->>WM: upsert-batch artifacts AG-->>UI: run complete
Same pack + same inputs always produces the same artifact sequence. The Astra Agent never consults capability packs or cap.* identifiers during this flow.

7. Seed Strategy

How capability seed data from capability-service is converted into skill seed data for skill-registry-service.

seed-strategy.mermaid
flowchart LR subgraph SOURCE["Source — capability seed files"] CAPFILE["cap.x.y.z\ntool_calls: [{tool: 'x', args_schema: {...}}]\ndescription: single sentence\nproduces_kinds: [cam.x.y]"] end subgraph CONVERT["Conversion"] R1["cap. → sk. prefix"] R2["tool_calls[0].tool → execution.tool_name\n(flatten array to string)"] R3["args_schema removed\n(resolved at runtime)"] R4["description enriched\n(LLM adds when-to-use context)"] R5["SKILL.md body generated\n(LLM from description + tool schema)"] R6["Multi-tool caps flagged\nfor manual split"] end subgraph TARGET["Target — skill seed files"] SKILLFILE["sk.x.y.z\nexecution.tool_name: 'x'\ndescription: enriched when+what\nproduces_kinds: [cam.x.y]\nSKILL.md body: generated"] end SOURCE --> CONVERT --> TARGET style SOURCE fill:#FEF3C7,stroke:#D97706 style CONVERT fill:#F1F5F9,stroke:#64748B style TARGET fill:#D1FAE5,stroke:#059669
Capabilities with multiple tool_calls entries are flagged during conversion as requiring manual review — each tool becomes a separate sk.* skill. The cam.* produces_kinds are copied verbatim — no changes to artifact kind identifiers.
